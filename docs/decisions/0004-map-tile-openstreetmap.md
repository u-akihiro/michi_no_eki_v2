# 地図タイルに OpenStreetMap 公式を継続採用する

## 状況

道の駅v2のMVPで、全国の道の駅を地図上にプロットする機能を実装する。地図ライブラリは Leaflet + react-leaflet を採用（旧repo踏襲）。

地図タイルの配信元を選定する必要がある。旧repoは OpenStreetMap 公式サーバ（`tile.openstreetmap.org`）を採用しており、旧repoADR-0005で「本番トラフィックが拡大した際は Stadia Maps・MapTiler などのCDNタイルプロバイダーへの移行を検討する」と明記されていた。

## 決定

**OpenStreetMap公式タイル（`tile.openstreetmap.org`）を継続採用する。**

ただし、以下の移行判断基準を明文化し、条件に達した時点で別ADRを起票して移行判断する。

### 移行検討トリガー

以下のいずれかに該当した場合、CDNタイルプロバイダーへの移行を検討する:

1. 月間タイルリクエスト数が10万件を超える見込みに達した
2. OSMFからレート制限・ブロックの通知や兆候（403/429の頻発）を検知した
3. サービスがSNS等で拡散され、トラフィックが急増した
4. 商用化・広告表示等の収益化を開始した

### 移行先候補

- **MapTiler**（OSMベース、月10万リクエスト無料枠）
- **Stadia Maps**（月20万リクエスト無料枠、商用は有料）
- **地理院タイル**（日本国内特化、無料、APIキー不要）

## 理由

### OSM公式を継続採用する理由

1. **今回のプロジェクトの主眼は「AWSからCloudflareへの再構築」**
   - 地図タイルの意思決定は本プロジェクトの主目的と直交
   - 移行スコープをインフラとコード刷新に集中させる
2. **旧repoでの実運用実績**
   - 既に運用され問題なく機能している選択
   - 旧ADR-0005で将来移行を明文化済み、その意思決定を引き継ぐ
3. **開発コストが最小**
   - APIキー不要
   - Cloudflare Secretsへの登録・環境変数管理が発生しない
   - Wrangler開発時の設定が簡素
4. **MVP段階のトラフィックでは実質問題が起きにくい**
   - 個人開発〜小規模利用のフェーズでは規約上の懸念が顕在化しない

### OSM公式のリスクを認識する

OSM Tile Usage Policy（https://operations.osmfoundation.org/policies/tiles/）には以下の制約がある:

- Heavy use（数値は非公開）は禁止
- Bulk download禁止
- 有効なUser-Agentと帰属表示が必須
- エンフォースメント（IPブロック等）は運営裁量

「静かに使っている限りは動くが、成功したらブロックされうる」構造であることを前提に運用する。

### 必須の運用ルール

1. **帰属表示**: 地図上に `© OpenStreetMap contributors` を必ず表示する
2. **タイルのキャッシュ**: ブラウザキャッシュ・CDNキャッシュを可能な限り効かせる
3. **監視**: タイル配信の失敗（403/429/タイムアウト）を検知する仕組みを検討する（本ADR時点では未実装、必要時に別Issue化）

### 却下した選択肢

- **地理院タイル（国土地理院）を最初から採用**
  - 日本国内サービスに最適・規約明確・無料という魅力はあるが、旧repoの選択を尊重して踏襲を優先
- **MapTilerを最初から採用**
  - 将来移行を先取りする案。ただしMVP段階でAPIキー管理を導入する意義が薄い
- **自前タイル配信（Self-hosted）**
  - MVP段階では過剰。ストレージ・配信コストが実質発生
