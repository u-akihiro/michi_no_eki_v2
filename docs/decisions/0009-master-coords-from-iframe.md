# マスターデータの緯度経度を michi-no-eki.jp 埋め込み iframe から取得する

## 状況

道の駅マップ（MVP 機能A）には各駅の緯度経度が必須である。michi-no-eki.jp の一覧・詳細ページからは名称・都道府県・住所・公式URLは取れるが、座標は直接のフィールドとしては提供されない。

旧repo（u-akihiro/michi_no_eki）は当初、住所文字列をデジタル庁 abr-geocoder でジオコーディングして座標を得ていたが、道の駅が立地しがちな郊外・山間部の「字〇〇」系住所では `machiaza`（町大字中心）レベルまでしか解決できず、実測で km 級誤差の駅が約20%発生した（旧repo ADR-0026）。最終的に旧repo は abr-geocoder を廃止し、詳細ページの Google Maps iframe から座標を取る方式へ移行している。

## 決定

各駅の緯度経度は、**詳細ページに埋め込まれた Google Maps iframe の `src` クエリ `q=lat,lng`**（`.../maps/embed/v1/place?q=36.1339762,137.1662097&key=...`）から抽出する。外部ジオコーディング API・API キーは使わない。

堅牢化（旧repo の実装から移植する）:

- **iframe 欠落の hard fail**: 座標を抽出できない駅が1件でもあれば非0終了する（`--allow-missing-coords` で上書き可）。
- **日本範囲外の検知**: `lat 20–46 / lon 122–154` の範囲外は「欠落」扱いに丸め、同じゲートで捕捉する。
- **複数 iframe の divergence 検知**: 同一ページに複数の地図 iframe があり座標が一致しない場合のみ warning。同一座標の重複埋め込みは偽陽性にしない。
- **回帰防止テスト**: `extractMapCoords` をテーブルテストで固定（クエリ順序揺れ・複数 iframe・範囲外・単語境界誤マッチ）。

## 理由

- **精度**: 公式サイトが地図表示のために埋め込む座標なので、実際の施設位置を指す。住所解決の曖昧さが入らない。
- **網羅性**: 旧repo の実績で iframe 欠落は0件。
- **取得コスト**: スクレイプ済み HTML から抽出でき、別バッチ（ジオコーダ）が不要。
- **依存**: 外部 API キー・有償ライセンスが不要。

### 却下した選択肢

- **abr-geocoder 継続**: 住所ベースである以上、郊外住所の精度の天井を超えられない。
- **国土地理院ジオコーディング API**: 同じく住所ベースで精度上限は同程度。
- **マップコード→座標変換**: デンソー独自仕様で公式変換は有償・非公式実装はライセンスがグレー。

### 受容したリスク（単一障害点）

iframe 一本化のため、michi-no-eki.jp 側で iframe URL 形式（`maps/embed/v1/place?q=lat,lng`）や埋め込み自体が変わると全駅で座標取得が落ちる。これは上記の hard fail ゲートで **silent failure を防ぐ**ことで緩和する（落ちたら気づける）。
